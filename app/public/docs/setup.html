Database Setup Guide

How to set up your own EntEdit database with GraphDB

← Back to Cataloguing Guide

1. Overview

EntEdit uses a SPARQL-compatible graph database to store and query RDF data. The recommended database is Ontotext GraphDB, which is included in the Docker Compose setup. This guide walks through setting up the database from scratch using Docker, and explains how to load your own data.

If you just want to try EntEdit, you can skip this guide and use the demo database at http://dijon.idi.ntnu.no:8080/repositories/EntEdit.

2. Prerequisites

  • Docker and Docker Compose — installed and running on your machine.
  • The EntEdit repository cloned from Git.
Tip: Docker Desktop (available for Windows, macOS, and Linux) includes both Docker and Docker Compose.

3. Starting the Docker environment

From the root of the EntEdit repository, run:

docker compose up -d

This starts three services:

Service URL Purpose
Web application http://localhost/entedit/ The EntEdit user interface (served by nginx)
GraphDB http://localhost:7200 The graph database (GraphDB Workbench for administration)
Initialization — A one-time service that creates the repository and imports data

When using the Docker setup, configure EntEdit with the endpoint URL http://localhost/graphdb/repositories/EntEdit. The nginx proxy forwards requests to GraphDB, which avoids cross-origin (CORS) issues.

Note: The first time you run docker compose up, the initialization service will create the GraphDB repository, import vocabulary files and test data, and set up the full-text search indexes. This may take a minute or two.

4. What gets initialized

The initialization service performs three phases:

  1. Repository creation — Creates a GraphDB repository named EntEdit with RDFS-Plus reasoning enabled.
  2. Data import — Loads all RDF files from two directories:
    • database/types/ — Vocabulary and ontology files (RDA properties, class definitions, controlled value lists)
    • database/testdata/ — Sample bibliographic entities (works, expressions, manifestations, agents)
  3. SPARQL updates — Executes setup queries from database/graphdb/, including the Lucene full-text search connector definitions that power the Search tab.

5. Repository structure

The relevant directories in the EntEdit repository:

EntEdit/
├── database/
│ ├── types/ ← Vocabulary files (loaded into GraphDB)
│ ├── testdata/ ← Sample RDF entities
│ ├── graphdb/ ← SPARQL connector definitions
│ └── inference rules/← GraphDB reasoning rules
├── docker/
│ ├── graphdb/ ← Repository config + init script
│ └── nginx/ ← Web server + proxy config
└── docker-compose.yml

6. Using your own data

Adding data before first launch

The simplest approach is to place your RDF files in the database/testdata/ directory before running docker compose up for the first time. Supported formats:

  • .ttl — Turtle
  • .rdf — RDF/XML
  • .nt — N-Triples

Importing data into an existing database

If the database is already running, you can import data through the GraphDB Workbench:

  1. Open http://localhost:7200 in your browser.
  2. Go to Import → RDF in the left sidebar.
  3. Upload your RDF files and import them into the EntEdit repository.
Important: After importing new data, you may need to recreate the Lucene full-text search indexes for the new entities to appear in search results. The connector definitions are in database/graphdb/.

7. Setting up your own database from scratch

If you want to use EntEdit with your own GraphDB instance (outside the Docker setup), you need to load the vocabulary files and set up the full-text search connectors. This section covers the minimum steps required.

Step 1: Create a repository

In the GraphDB Workbench (http://localhost:7200), go to Setup → Repositories → Create new repository. Use the following settings:

  • Repository ID: EntEdit (or any name you prefer)
  • Ruleset: RDFS-Plus (Optimized)

All other settings can be left at their defaults.

Step 2: Import the vocabulary files

EntEdit requires the vocabulary files under database/types/ to be loaded into the database. These define the classes, properties, controlled values, and display configuration that the application relies on. The directory contains three subdirectories:

Directory Contents
database/types/entedit profile/ The EntEdit application profile — defines which classes and properties are active, their display order, and status. This is required.
database/types/rda vocabulary/ The RDA (Resource Description and Access) ontology — class and property definitions with labels in multiple languages.
database/types/term vocabularies/ Controlled value lists — languages, content types, media types, carrier types, work types.

Import all files from these directories via Import → RDF in the GraphDB Workbench, or use the GraphDB REST API.

Important: All three directories must be imported. Without the EntEdit profile, no classes or properties will appear in the application. Without the RDA vocabulary, labels will be missing. Without the term vocabularies, controlled value dropdowns will be empty.

Step 3: Set up the Lucene connectors

The Search tab in EntEdit relies on GraphDB's Lucene connector for full-text search. You need to execute the connector definition queries to create the search indexes. These are SPARQL UPDATE statements found in:

  • database/graphdb/expression.connector.sparql — Creates the full-text index for Expression search
  • database/graphdb/manifestation.connector.sparql — Creates the full-text index for Manifestation search

Execute these in the GraphDB Workbench under SPARQL → SPARQL Update, or via the SPARQL endpoint. After execution, the connectors will automatically index existing data and keep the index updated as new data is added.

Tip: If you skip this step, the Entity Browser will still work — you can browse and edit entities normally. Only the Search tab requires the Lucene connectors.

Step 4: Import your bibliographic data

With the vocabulary in place, you can now import your own bibliographic data (Works, Expressions, Manifestations, Agents) via GraphDB's import tools. The data should use the RDA vocabulary URIs defined in the vocabulary files. See the sample files in database/testdata/ for examples of the expected RDF structure.

8. Ontology requirements

EntEdit expects certain conventions in the ontology to determine which classes and properties to display and how to order them:

Predicate Purpose
entedit:status Must be set to an active status value for a class or property to appear in the application. Classes and properties without this predicate are hidden.
entedit:order A numeric value that controls the display order of properties within the editor. Lower values appear first.
rdfs:domain Standard RDFS. Used to determine which properties are available for a given class.
rdfs:range Standard RDFS. Used to determine what values are valid for object properties.
rdfs:label Used to display human-readable names for classes, properties, and entities. Should be provided in both English and Norwegian for full bilingual support.

The vocabulary files in database/types/ provide a working example of how these predicates are used with the RDA ontology. If you are building your own ontology, use these files as a reference.

9. Development setup

For local development of the EntEdit web application (without Docker for the frontend):

  1. Make sure you have Node.js 20+ installed.
  2. Start a GraphDB instance (either via Docker or a standalone installation) with the EntEdit repository.
  3. Install dependencies and start the dev server: 
    cd app
npm install
npm run dev
  4. Open http://localhost:5173 in your browser.

The Vite development server proxies /graphdb requests to http://localhost:7200, so you can use the endpoint URL http://localhost:7200/repositories/EntEdit or let the proxy handle it.

Tip: If you are only developing the frontend, you can start just the GraphDB service with docker compose up graphdb graphdb-init -d and run the web app locally with npm run dev.
