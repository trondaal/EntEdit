Oppsettguide for database

Slik setter du opp din egen EntEdit-database med GraphDB

← Tilbake til katalogiseringsveiledningen

1. Oversikt

EntEdit bruker en SPARQL-kompatibel grafdatabase for lagring og spørring av RDF-data. Den anbefalte databasen er Ontotext GraphDB, som er inkludert i Docker Compose-oppsettet. Denne guiden forklarer hvordan du setter opp databasen fra bunnen av med Docker, og hvordan du laster inn egne data.

Hvis du bare vil prøve EntEdit, kan du hoppe over denne guiden og bruke demodatabasen på http://dijon.idi.ntnu.no:8080/repositories/EntEdit.

2. Forutsetninger

  • Docker og Docker Compose — installert og kjørende på maskinen din.
  • EntEdit-repositoriet klonet fra Git.
Tips: Docker Desktop (tilgjengelig for Windows, macOS og Linux) inkluderer både Docker og Docker Compose.

3. Starte Docker-miljøet

Fra rotkatalogen til EntEdit-repositoriet, kjør:

docker compose up -d

Dette starter tre tjenester:

Tjeneste URL Formål
Webapplikasjon http://localhost/entedit/ EntEdit-brukergrensesnittet (servert av nginx)
GraphDB http://localhost:7200 Grafdatabasen (GraphDB Workbench for administrasjon)
Initialisering — En engangstjeneste som oppretter repositoriet og importerer data

Når du bruker Docker-oppsettet, konfigurer EntEdit med endepunkt-URL-en http://localhost/graphdb/repositories/EntEdit. nginx-proxyen videresender forespørsler til GraphDB, noe som unngår kryssopprinnelsesproblemer (CORS).

Merk: Første gang du kjører docker compose up, vil initialiseringstjenesten opprette GraphDB-repositoriet, importere vokabularfiler og testdata, og sette opp fulltekstsøkeindeksene. Dette kan ta et minutt eller to.

4. Hva som initialiseres

Initialiseringstjenesten utfører tre faser:

  1. Oppretting av repositorium — Oppretter et GraphDB-repositorium med navnet EntEdit med RDFS-Plus-resonnering aktivert.
  2. Dataimport — Laster inn alle RDF-filer fra to kataloger:
    • database/types/ — Vokabular- og ontologifiler (RDA-egenskaper, klassedefinisjoner, kontrollerte verdilister)
    • database/testdata/ — Eksempler på bibliografiske entiteter (verk, uttrykk, manifestasjoner, agenter)
  3. SPARQL-oppdateringer — Kjører oppsettspørringer fra database/graphdb/, inkludert Lucene-fulltekstsøkekoblingsdefinisjoner som driver Søk-fanen.

5. Katalogstruktur

De relevante katalogene i EntEdit-repositoriet:

EntEdit/
├── database/
│ ├── types/ ← Vokabularfiler (lastes inn i GraphDB)
│ ├── testdata/ ← Eksempel-RDF-entiteter
│ └── graphdb/ ← SPARQL-koblingsdefinisjoner
├── docker/
│ ├── graphdb/ ← Repositoriumkonfigurasjon + init-skript
│ └── nginx/ ← Webserver- og proxy-konfigurasjon
└── docker-compose.yml

6. Bruke egne data

Legge til data før første oppstart

Den enkleste fremgangsmåten er å plassere RDF-filene dine i katalogen database/testdata/ før du kjører docker compose up for første gang. Støttede formater:

  • .ttl — Turtle
  • .rdf — RDF/XML
  • .nt — N-Triples

Importere data til en eksisterende database

Hvis databasen allerede kjører, kan du importere data gjennom GraphDB Workbench:

  1. Åpne http://localhost:7200 i nettleseren din.
  2. Gå til Import → RDF i sidemenyen til venstre.
  3. Last opp RDF-filene dine og importer dem til EntEdit-repositoriet.
Viktig: Etter import av nye data kan det være nødvendig å gjenskape Lucene-fulltekstsøkeindeksene for at de nye entitetene skal dukke opp i søkeresultatene. Koblingsdefinisjonene finnes i database/graphdb/.

7. Sette opp en egen database fra bunnen av

Hvis du vil bruke EntEdit med din egen GraphDB-instans (utenfor Docker-oppsettet), må du laste inn vokabularfilene og sette opp fulltekstsøkekoblingene. Denne seksjonen dekker de nødvendige minimumsstegene.

Steg 1: Opprett et repositorium

I GraphDB Workbench (http://localhost:7200), gå til Setup → Repositories → Create new repository. Bruk følgende innstillinger:

  • Repository ID: EntEdit (eller et annet navn du foretrekker)
  • Ruleset: RDFS-Plus (Optimized)

Alle andre innstillinger kan stå på standardverdiene.

Steg 2: Importer vokabularfilene

EntEdit krever at vokabularfilene under database/types/ er lastet inn i databasen. Disse definerer klassene, egenskapene, kontrollerte verdier og visningskonfigurasjonen som applikasjonen er avhengig av. Katalogen inneholder tre undermapper:

Katalog Innhold
database/types/entedit_profile/ EntEdit-applikasjonsprofilen — definerer hvilke klasser og egenskaper som er aktive, visningsrekkefølgen og status. Denne er påkrevd.
database/types/rda_vocabulary/ RDA-ontologien (Resource Description and Access) — klasse- og egenskapsdefinisjoner med etiketter på flere språk.
database/types/term_vocabularies/ Kontrollerte verdilister — språk, innholdstyper, medietyper, bærertyper, verktyper.

Importer alle filer fra disse katalogene via Import → RDF i GraphDB Workbench, eller bruk GraphDB REST API.

Viktig: Alle tre katalogene må importeres. Uten EntEdit-profilen vil ingen klasser eller egenskaper vises i applikasjonen. Uten RDA-vokabularet vil etiketter mangle. Uten termvokabularene vil nedtrekkslistene for kontrollerte verdier være tomme.

Steg 3: Sett opp Lucene-koblingene

Søk-fanen i EntEdit er avhengig av GraphDBs Lucene-kobling for fulltekstsøk. Du må kjøre koblingsdefinisjonsspørringene for å opprette søkeindeksene. Dette er SPARQL UPDATE-setninger som finnes i:

  • database/graphdb/expression.connector.sparql — Oppretter fulltekstindeksen for uttrykkssøk
  • database/graphdb/manifestation.connector.sparql — Oppretter fulltekstindeksen for manifestasjonssøk

Kjør disse i GraphDB Workbench under SPARQL → SPARQL Update, eller via SPARQL-endepunktet. Etter kjøring vil koblingene automatisk indeksere eksisterende data og holde indeksen oppdatert når nye data legges til.

Tips: Hvis du hopper over dette steget, vil entitetleseren fortsatt fungere — du kan bla gjennom og redigere entiteter som normalt. Bare Søk-fanen krever Lucene-koblingene.

Steg 4: Importer bibliografiske data

Med vokabularet på plass kan du nå importere dine egne bibliografiske data (verk, uttrykk, manifestasjoner, agenter) via GraphDBs importverktøy. Dataene bør bruke RDA-vokabularets URI-er som er definert i vokabularfilene. Se eksempelfilene i database/testdata/ for eksempler på forventet RDF-struktur.

8. Ontologikrav

EntEdit forventer visse konvensjoner i ontologien for å bestemme hvilke klasser og egenskaper som skal vises, og hvordan de skal sorteres:

Predikat Formål
entedit:status Må settes til en aktiv statusverdi for at en klasse eller egenskap skal vises i applikasjonen. Klasser og egenskaper uten dette predikatet er skjulte.
entedit:order En numerisk verdi som styrer visningsrekkefølgen til egenskaper i editoren. Lavere verdier vises først.
rdfs:domain Standard RDFS. Brukes til å bestemme hvilke egenskaper som er tilgjengelige for en gitt klasse.
rdfs:range Standard RDFS. Brukes til å bestemme hvilke verdier som er gyldige for objektegenskaper.
rdfs:label Brukes til å vise lesbare navn for klasser, egenskaper og entiteter. Bør oppgis på både engelsk og norsk for full tospråklig støtte.

Vokabularfilene i database/types/ gir et fungerende eksempel på hvordan disse predikatene brukes med RDA-ontologien. Hvis du bygger din egen ontologi, kan du bruke disse filene som referanse.

9. Utviklingsoppsett

For lokal utvikling av EntEdit-webapplikasjonen (uten Docker for frontenden):

  1. Sørg for at du har Node.js 20+ installert.
  2. Start en GraphDB-instans (enten via Docker eller en frittstående installasjon) med EntEdit-repositoriet.
  3. Installer avhengigheter og start utviklingsserveren: 
    cd app
npm install
npm run dev
  4. Åpne http://localhost:5173 i nettleseren din.

Vite-utviklingsserveren videresender /graphdb-forespørsler til http://localhost:7200, så du kan bruke endepunkt-URL-en http://localhost:7200/repositories/EntEdit eller la proxyen håndtere det.

Tips: Hvis du bare utvikler frontenden, kan du starte kun GraphDB-tjenesten med docker compose up graphdb graphdb-init -d og kjøre webapplikasjonen lokalt med npm run dev.
